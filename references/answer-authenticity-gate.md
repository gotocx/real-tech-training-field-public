# Answer Authenticity Gate

这个门禁用于防止用户把未经消化的生成内容当作训练完成。目标不是羞辱用户，而是戳破自欺，让答案回到真实面试和真实项目标准。

## 触发场景

- 用户粘贴代码、测试、题解、八股回答、项目总结。
- 用户问“这样行不行”“能不能算过”“这个答案怎么样”。
- 用户给出的内容看起来完整，但没有证明自己理解。

## 高风险信号

- 只有正常路径，没有失败路径。
- 注释很多，但都在解释显而易见的语句。
- 没有边界条件、复杂度、取舍或反例。
- 用词平滑完整，但不能回答为什么这样测、为什么这样实现。
- 测试只证明样例能过，不能打破错误实现。

## 必须输出的批评格式

1. 先判定：`这个答案看起来像未经消化的生成内容，真实面试中过不了。`
2. 再指出具体缺陷，不使用泛泛评价。
3. 最后给出修补任务，要求用户补能证明能力的内容。

## 典型例子

用户粘贴 two sum 的单个正常用例测试，并写了大量解释性注释。这不能算通过。

应指出：

- 只测了 happy path，不能证明无解、重复值、负数、同一元素不能复用等边界。
- `result == [0, 1] or result == [1, 0]` 只是放宽顺序，没有验证返回下标是否真的对应目标和。
- 注释解释了“正常情况”这类显而易见信息，但没有说明测试为什么能打破错误实现。
- 真实面试中，这种测试会被认为只会跟着样例走，缺少验证能力。

修补任务：

- 补无解用例，要求返回空结果或约定错误。
- 补重复值用例，例如 `[3, 3]` 和目标 `6`。
- 补负数或零的用例。
- 补一个能抓出“同一元素被复用”的错误实现的用例。
- 说明每个测试对应要防住哪类错误。
